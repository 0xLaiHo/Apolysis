# v0.2.0 Release Publication

Use this checklist after the signed release artifact dry run has passed and
before pushing the final `v0.2.0` tag. It keeps the final publication path
reviewed, signed, and easy to roll back without treating a workflow-dispatch
artifact as a public GitHub Release.

The latest signed dry run is workflow run `28694166781` for
`v0.2.0-signed-dry-run.20260704035609`. That run built the Linux CLI, built the
CO-RE BPF object, signed the release manifest with AWS KMS, packaged release
signing evidence, and uploaded the Actions artifact bundle.

## Required Repository Settings

Confirm these repository settings before review:

- `ProductionHardening_AWS_ROLE_TO_ASSUME`: GitHub secret containing the AWS
  IAM role ARN trusted by GitHub OIDC.
- `ProductionHardening_AWS_KMS_KEY_ID`: GitHub secret containing the AWS KMS
  signing key id, alias, or ARN.
- `ProductionHardening_AWS_REGION`: GitHub variable containing the AWS region.
- `APOLYSIS_RELEASE_SIGNING_EVIDENCE_RUN_ID`: GitHub variable currently set to
  `28694166781`.

Do not print secret values. Only confirm the names exist and that
`APOLYSIS_RELEASE_SIGNING_EVIDENCE_RUN_ID` points at the retained signed dry-run
evidence.

Also confirm the AWS IAM role trust policy authorizes the tag publication
subject before pushing the final tag. A signed workflow-dispatch dry run from
`main` proves the branch subject, but it does not prove the tag subject. The
trust policy must allow GitHub OIDC `sts:AssumeRoleWithWebIdentity` for either
the exact final release subject:

```text
repo:0xLaiHo/Apolysis:ref:refs/tags/v0.2.0
```

or a bounded release-tag pattern:

```text
repo:0xLaiHo/Apolysis:ref:refs/tags/v*
```

Keep the trust scope limited to this repository and release tags.

## Release Branch Review

Start from the dedicated release branch:

```bash
git switch release/v0.2.0
git pull --ff-only origin release/v0.2.0
```

Run the no-secret local release gates:

```bash
make test-release-publication-readiness
make test-release-validation-ci
make test-release-artifacts
make test-release-signing
make test-signed-release-artifact-dry-run
git diff --check
```

Open the release pull request:

```bash
gh pr create \
  --repo 0xLaiHo/Apolysis \
  --base main \
  --head release/v0.2.0 \
  --title "release: prepare v0.2.0 publication" \
  --body-file target/release-v0.2.0-pr-body.md
```

Do not push the final tag before release branch review passes. The review must
include the signed dry-run evidence, the no-secret gate results, and the
explicit statement that tag publication will invoke the `Release Artifacts`
workflow with AWS OIDC/KMS signing.

## Final Tag

After the release branch is approved and merged:

```bash
git switch main
git pull --ff-only origin main
git tag -a v0.2.0 -m "Apolysis v0.2.0"
git push origin v0.2.0
```

Find and watch the tag-triggered workflow:

```bash
run_id="$(gh run list \
  --repo 0xLaiHo/Apolysis \
  --workflow release-artifacts.yml \
  --limit 10 \
  --json databaseId,headBranch \
  --jq '.[] | select(.headBranch == "v0.2.0") | .databaseId' \
  | head -n 1)"

test -n "$run_id"
gh run watch "$run_id" --repo 0xLaiHo/Apolysis --exit-status
```

The workflow must build the release package, sign the manifest, package signing
evidence, upload the Actions artifact bundle, create or update the GitHub
Release, and upload release assets.

## Published Release Verification

Download and verify the public GitHub Release assets:

```bash
mkdir -p target/release-v0.2.0
gh release download v0.2.0 \
  --repo 0xLaiHo/Apolysis \
  --dir target/release-v0.2.0

cd target/release-v0.2.0
sha256sum -c apolysis-v0.2.0-x86_64-unknown-linux-gnu.tar.gz.sha256
jq '{release_signing_ready, provider, release_manifest_sha256}' \
  apolysis-release-signing-manifest.json
jq '{source, provider, key_uri, signature_verified, release_manifest_sha256}' \
  apolysis-release-signing-evidence.json
gh release view v0.2.0 --repo 0xLaiHo/Apolysis
```

The release must include:

- `apolysis-v0.2.0-x86_64-unknown-linux-gnu.tar.gz`
- `apolysis-v0.2.0-x86_64-unknown-linux-gnu.tar.gz.sha256`
- `apolysis-release-manifest.json`
- `apolysis-release-signing-manifest.json`
- `apolysis-release-signing-evidence.json`
- `apolysis-release-signing-report.json`
- `apolysis-regulated-release-signing-evidence-report.json`

Expected signed state:

- `release_signing_ready` is `true`.
- `provider` is `cloud_kms`.
- `key_uri` starts with `awskms://`.
- `signature_verified` is `true`.
- `release_manifest_sha256` matches the SHA-256 of
  `apolysis-release-manifest.json`.

## Rollback Boundary

If the tag-triggered workflow fails before public assets are uploaded, delete
the failed tag only after recording the run id and failure reason:

```bash
git push origin :refs/tags/v0.2.0
git tag -d v0.2.0
```

If the GitHub Release exists with incomplete assets, preserve the failed run id,
delete only the incomplete release draft/assets after review, fix the release
branch, and publish a new tag attempt from a reviewed `main`.

If the workflow fails at `Configure AWS credentials for release signing` with
`Could not assume role with OIDC: Not authorized to perform
sts:AssumeRoleWithWebIdentity`, the AWS role trust policy is missing the tag
subject. Delete the failed tag as above, update the role trust to allow the
`refs/tags/v0.2.0` subject or bounded `refs/tags/v*` release pattern, then
repeat the final tag flow from reviewed `main`.
